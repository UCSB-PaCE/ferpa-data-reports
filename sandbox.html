Before you write (or generate) a single line

Get your LLM Sandbox key (UCSB) #

Everything here needs two things from UCSB's LLM Sandbox: a base URL and an API key. There is no cost, this is the campus gateway. Here is how to get them.

Publish a bot and use its API (available now)

Build a bot in LLM Sandbox (llmsandbox.cloud.ucsb.edu), give it instructions and, if you like, knowledge (files or a URL). Then from Settings → My Bots open the bot's three-dots menu and choose API Publish Settings.

Publishing deploys a small REST API for that bot and gives you both an API Endpoint (the URL) and an API Key on one screen, each with a copy button, so you do not have to hunt for the URL separately. Copy both; the key stays hidden until you click Show. This route is open to anyone with LLM Sandbox access, and it is exactly the endpoint plus key that powers the "Ask the assistant" widget on this page.

Treat the key like a password Anyone with your key can call the API as you, and the usage counts against your quota. Store it in a secrets manager (Colab Secrets, an environment variable, or a vault), never in a cell, a commit, or a public page. If a key might have leaked, revoke it and generate a new one.
Bot API vs raw model calls A published bot API talks to a bot you configured (its own prompt and knowledge), so it is a different shape from the raw, OpenAI-compatible model calls in the code snippets below. Reach for the bot API when you want a hosted assistant; reach for the direct key below when you want to call the model yourself.

Get a direct key and your base URL (developer preview)

For the raw, OpenAI-compatible calls the pipeline code uses, LLM Sandbox has a developer console at llmdev.cloud.ucsb.edu. Under the menu (top left) → Settings → My API Keys you can see your Base URL (paste it as the base_url in any OpenAI-compatible client, or as LLM_SANDBOX_BASE_URL in Colab Secrets) and a Generate new key button for a direct, account-level key. This is the key style the pipeline code and the Colab notebook here use.

Currently in preview The developer console is in preview for people on the LLM Sandbox dev program, and it will roll out more broadly over time. If you do not have access to it yet, ask the LLM Sandbox team, and use the bot route above in the meantime.
Two values, the base URL and a key, unlock everything here. Grab them once, store the key as a Secret, and you never touch it again.

The centerpiece

LLM Sandbox API reference #

UCSB's LLM Sandbox exposes an OpenAI-compatible Chat Completions API in front of cloud foundation models (Anthropic Claude on Amazon Bedrock, among others). "OpenAI-compatible" is the key fact: any OpenAI SDK or any tool that speaks the Chat Completions shape works against it, you only change the base URL and the auth header. Everything in this section generalizes to any gateway built on the same convention (LiteLLM, Bedrock Access Gateway, vLLM's OpenAI server, Azure OpenAI, etc.).

Base URL & authentication #

SettingValueWhere it comes from
Base URLhttps://<your-llm-sandbox-host>/v1Your gateway admin (at UCSB, the LLM Sandbox team). Ends in /v1 like the OpenAI API.
Auth headerAuthorization: Bearer <KEY> or x-api-key: <KEY>Either one authenticates. OpenAI SDKs send the bearer automatically; x-api-key is handy for raw HTTP.
Modelclaude-v4.5-haikuOne of many. List the live catalog with GET {BASE_URL}/models (see below).
Content typeapplication/jsonStandard for all requests.
Good news: your key works two ways LLM Sandbox accepts your key as a standard Authorization: Bearer <KEY> (exactly what the OpenAI SDKs already send) or as an x-api-key: <KEY> header. Either one authenticates, so a normal OpenAI client works with just a base_url and api_key, no header surgery. The raw HTTP snippets use x-api-key because it is explicit; the bearer is equally valid. (Verified against the live gateway, Jun 2026.)

Endpoint: Chat Completions #

POST {BASE_URL}/chat/completions

Request body (the fields this pipeline uses; others from the OpenAI spec are accepted):

{
  "model": "claude-v4.5-haiku",
  "max_tokens": 1024,
  "messages": [
    { "role": "system", "content": "You are a helpful assistant specialized in analyzing feedback." },
    { "role": "user",   "content": "[1] Great instructor...\n[2] Office hours were hard to reach..." },
    { "role": "user",   "content": "Generate a keyword summary and a cited insight..." }
  ]
}

Response body (OpenAI shape, read choices[0].message.content and usage):

{
  "id": "chatcmpl-...",
  "object": "chat.completion",
  "model": "claude-v4.5-haiku",
  "choices": [
    { "index": 0,
      "message": { "role": "assistant",
                   "content": "Keyword Summary: Clarity (Green), Office Hours (Orange);Feedback Insight: ..." },
      "finish_reason": "stop" }
  ],
  "usage": { "prompt_tokens": 1060, "completion_tokens": 254, "total_tokens": 1314 }
}

The usage object is how you meter cost. On LLM Sandbox the budget is tokens, not dollars, total_tokens is the number that matters. A typical analysis is ≈ 1,060 in + 254 out ≈ 1.3K tokens.

Common parameters & models #

FieldTypeNotes
modelstringRequired. A model ID from the gateway's catalog.
messagesarrayRequired. {role, content} objects; roles are system, user, assistant.
max_tokensintegerCap on the completion length. this pipeline uses 1024, enough for a keyword line + two short paragraphs.
temperaturenumberOptional. Lower (0 to 0.4) for consistent, reproducible summaries; the cache makes determinism cheap.
streambooleanOptional. For batch report generation you almost always want false. For a live chat UI, set true and read the server-sent-event token chunks as they arrive.
Picking a model Summarizing short free-text into a few keywords and a paragraph is a small, cheap-model job. A Haiku-class model is the right default: fast, low token cost, and more than capable. Reserve larger models for genuinely harder reasoning, they cost more tokens for no benefit here.
The live model catalog (verified Jun 2026) GET {BASE_URL}/models on UCSB's gateway returns a multi-provider list. Anthropic: claude-v4.6-sonnet, claude-v4.6-opus, claude-v4.5-haiku, claude-v3.7-sonnet, claude-v3.5-haiku, claude-v3-opus. Amazon: amazon-nova-pro, -lite, -micro. Also llama-4-maverick-17b-instruct, llama-4-scout-17b-instruct, llama3-3-70b-instruct, mistral-large-2, mixtral-8x7b-instruct, deepseek-r1, qwen3-32b, and OpenAI-named aliases like gpt-4o. Always pull the current list from /models; the catalog grows over time.

Errors & retries #

Because it's OpenAI-compatible, errors come back as HTTP status codes with a JSON body. Handle them the way you'd handle the OpenAI API:

StatusMeaningWhat to do
401 / 403Key missing, wrong, or revokedSend a current key as Authorization: Bearer or x-api-key. Don't retry.
404Unknown model or wrong pathVerify the model ID and that the base URL ends in /v1.
429Rate / quota limitBack off and retry with exponential delay. Caching reduces how often you hit this.
5xxGateway / upstream errorRetry a few times with backoff; then log and skip that record (don't fail the whole run).

The production rule here: a failed analysis returns an "Error in analysis: ..." string and is not cached, so the next run retries it cleanly. One bad record never blocks the rest of the batch.

If you can call the OpenAI API, you can call LLM Sandbox. Point your client at the base URL, pass your key (as Bearer or x-api-key), and every OpenAI tool, SDK, and snippet just works.

Copy, paste, adapt

System-agnostic snippets #

Here is the same call in four flavors. Each takes a list of free-text comments and returns a keyword summary + cited insight. The base URL, key, and model always come from the environment / a secrets store, never hard-code them.

No header surgery needed Your key authenticates as either Authorization: Bearer <KEY> (what the OpenAI SDK sends automatically) or x-api-key: <KEY>, so a normal OpenAI client just works. These snippets use x-api-key because it is explicit in raw HTTP. If a call fails, it is almost always a row in the cookbook below. 
# pip install openai
import os
from openai import OpenAI

client = OpenAI(
    base_url=os.environ["LLM_SANDBOX_BASE_URL"],   # https://<host>/v1
    api_key=os.environ["LLM_SANDBOX_API_KEY"],     # the SDK sends it as Bearer; the gateway accepts it
)

def analyze(comments):
    numbered = "\n".join(f"[{i}] {c}" for i, c in enumerate(comments, 1))
    resp = client.chat.completions.create(
        model=os.environ.get("LLM_SANDBOX_MODEL", "claude-v4.5-haiku"),
        max_tokens=1024,
        messages=[
            {"role": "system", "content": SYSTEM_PROMPT},
            {"role": "user",   "content": numbered},
            {"role": "user",   "content": TASK_PROMPT},
        ],
    )
    print("tokens used:", resp.usage.total_tokens)   # your real budget
    return resp.choices[0].message.content
# pip install requests, no SDK, pure HTTP
import os, requests

def analyze(comments):
    numbered = "\n".join(f"[{i}] {c}" for i, c in enumerate(comments, 1))
    r = requests.post(
        os.environ["LLM_SANDBOX_BASE_URL"] + "/chat/completions",
        headers={
            "x-api-key": os.environ["LLM_SANDBOX_API_KEY"],
            "Content-Type": "application/json",
        },
        json={
            "model": os.environ.get("LLM_SANDBOX_MODEL", "claude-v4.5-haiku"),
            "max_tokens": 1024,
            "messages": [
                {"role": "system", "content": SYSTEM_PROMPT},
                {"role": "user",   "content": numbered},
                {"role": "user",   "content": TASK_PROMPT},
            ],
        },
        timeout=60,
    )
    r.raise_for_status()
    data = r.json()
    return data["choices"][0]["message"]["content"]
// Node 18+ has fetch built in. No dependencies.
async function analyze(comments) {
  const numbered = comments.map((c, i) => `[${i + 1}] ${c}`).join("\n");
  const res = await fetch(process.env.LLM_SANDBOX_BASE_URL + "/chat/completions", {
    method: "POST",
    headers: {
      "x-api-key": process.env.LLM_SANDBOX_API_KEY,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      model: process.env.LLM_SANDBOX_MODEL ?? "claude-v4.5-haiku",
      max_tokens: 1024,
      messages: [
        { role: "system", content: SYSTEM_PROMPT },
        { role: "user",   content: numbered },
        { role: "user",   content: TASK_PROMPT },
      ],
    }),
  });
  if (!res.ok) throw new Error(`Sandbox ${res.status}`);
  const data = await res.json();
  return data.choices[0].message.content;
}

curl "$LLM_SANDBOX_BASE_URL/chat/completions" \

    ]
  }'
On Windows The cURL tab uses POSIX single-quoting and $VAR expansion (macOS, Linux, Git Bash). In Windows cmd or PowerShell the quoting and variables differ; use Git Bash or WSL, or reach for the Python or Node snippet instead.
The two prompts These snippets reference SYSTEM_PROMPT and TASK_PROMPT. Those are this pipeline's feedback-summarization prompts; you supply your own for your task. The full guide shows the exact citation-grounded prompts you can lift verbatim.
Verified runnable: every snippet here was checked against the OpenAI Python SDK (1.102) and Node 22, and the call was confirmed live against UCSB's gateway. Change one thing, the base URL, and your existing OpenAI tooling works.

When it does not work

Common-errors cookbook #

Most failures calling an OpenAI-compatible gateway are auth or path issues, not your code. These are the real ones, with the fix. The snippets above are verified runnable, so if a call fails it is almost always a row below.

SymptomLikely causeFix
401 / 403 on every request.The key is missing, wrong, or revoked. (LLM Sandbox accepts the key as either Authorization: Bearer or x-api-key, so this is not a bearer-vs-header problem.)Send a current key in either header. Generate a fresh one under Settings → My API Keys if unsure. Do not retry a 401.
It works in the dev-console chat, but your API call returns 401.The console login and the /v1 API use different credentials. Your browser session does not authenticate API calls.Create a key under Settings → My API Keys and send it as Authorization: Bearer or x-api-key.
404 on POST /chat/completions.The base URL is missing /v1, or the path is doubled (the base URL already ends in /chat/completions).Base URL ends in /v1. The SDK appends /chat/completions for you; in requests / fetch / curl you append it once, so never put /chat/completions in the env var.
404 model_not_found even though auth works.The model id is not in this gateway's catalog, or it is misspelled.List the live ids with GET {BASE_URL}/models (verified working). Real ids include claude-v4.5-haiku, claude-v4.6-sonnet, amazon-nova-pro, and gpt-4o. Set it via LLM_SANDBOX_MODEL; do not hard-code.
OpenAIError: the api_key client option must be set at startup.api_key is None and no OPENAI_API_KEY is set. In Colab, userdata.get() returns None for a missing or disabled Secret.Pass a non-empty api_key. In Colab, add and enable all three Secrets (base URL, key, model).
stream: true hangs or returns nothing readable from choices.Streaming returns server-sent-event chunks that use delta, not a single message body; some proxies also buffer SSE.For batch reports keep stream: false and read choices[0].message.content. For a live UI, iterate the SSE delta and confirm nothing in the path buffers.
Intermittent 429 or 5xx in the middle of a batch.429 is a rate or token-quota limit; 5xx is a gateway or upstream error. Neither is an auth or syntax problem.Back off and retry on 429 / 5xx; do not retry 401 / 403 / 404. A warm cache cuts how often you hit 429.
Verified against the live gateway (Jun 2026) Checked directly on UCSB's LLM Sandbox: the Base URL ends in /v1; the key authenticates as either Authorization: Bearer or x-api-key; GET {BASE_URL}/models returns the live catalog (Claude v4.6 / v4.5, Amazon Nova, Llama 4, Mistral, DeepSeek, Qwen, and gpt-4o aliases); and a claude-v4.5-haiku chat completion returns the standard OpenAI shape. Two things still depend on your setup: whether max_tokens is required, and whether a long streaming response passes through unbuffered. Confirm those for your case.
