For developers, at any institution

Call a language model your organization controls

The pipeline only sends text to a model in one place: through a gateway your organization runs. This page is how to find that gateway, or get one, and how to call it. The API is the same one OpenAI made popular, so your existing tools work with two changes: the base URL and the key.

Start here

You need a gateway you control #

There is exactly one step in this whole pipeline where your data leaves your own code: the model call. That makes the question of where the call goes the most important decision you make. For anything sensitive, it has to go somewhere covered by agreements your institution already holds.

A consumer chatbot will not do for protected records. Read the terms of most public AI services and they keep the right to retain and train on what you send, which a university cannot agree to on a student's behalf. The answer is not to avoid AI. It is to send the text to a gateway you control.

What "a gateway" means An internal endpoint that speaks the same API as OpenAI, sitting in front of approved models. You get a base URL and a key. The data stays inside your boundary, and nothing is kept for training. Everything below works against any gateway built this way.

Don't have one yet? #

If your institution does not run a gateway, this is a conversation for IT or whoever owns data governance, not something to stand up alone on a personal account. The usual ways organizations provide one:

  • Azure OpenAI Service, under the institution's own tenant and data terms.
  • An Amazon Bedrock gateway, for example the open-source Bedrock Access Gateway. UCSB's gateway runs on this.
  • A self-hosted proxy such as LiteLLM or vLLM in front of approved models.

All three hand you a base URL and a key and speak the Chat Completions API, so the code here runs against any of them. The point is an endpoint your organization controls, not just any AI you can sign up for. At UCSB, that gateway is the LLM Sandbox.

The interface

How the API works #

"OpenAI-compatible" is the useful fact here: any OpenAI SDK, and any tool that speaks the Chat Completions shape, works against your gateway once you point it at the right base URL and key. The same applies to gateways built on LiteLLM, the Bedrock Access Gateway, vLLM's OpenAI server, Azure OpenAI, and others.

Base URL and authentication #

SettingValueWhere it comes from
Base URLhttps://<your-gateway-host>/v1Your gateway admin. Ends in /v1, like the OpenAI API.
Auth headerAuthorization: Bearer <KEY>The standard. Some gateways use x-api-key instead; see the note below.
Modela model id from your gateway's catalogList the live ids with GET {BASE_URL}/models.
Content typeapplication/jsonStandard for all requests.
Which auth header? Most gateways accept the standard Authorization: Bearer <KEY> that every OpenAI SDK already sends, so you change nothing but the base URL and the key. Some accept or require an x-api-key: <KEY> header instead. If a bearer call returns 401, try x-api-key. Your gateway's docs say which; the UCSB page documents LLM Sandbox, which happens to take either.

Endpoint: Chat Completions #

POST {BASE_URL}/chat/completions

Request body (the fields most pipelines use; the rest of the OpenAI spec is accepted):

{
  "model": "your-model-id",
  "max_tokens": 1024,
  "messages": [
    { "role": "system", "content": "You summarize free-text responses." },
    { "role": "user",   "content": "[1] First response...\n[2] Second response..." },
    { "role": "user",   "content": "List the main themes, each with the [n] that support it." }
  ]
}

Response body (OpenAI shape: read choices[0].message.content and usage):

{
  "choices": [
    { "index": 0,
      "message": { "role": "assistant", "content": "Theme 1: ... [1][4]. Theme 2: ... [2]." },
      "finish_reason": "stop" }
  ],
  "usage": { "prompt_tokens": 1060, "completion_tokens": 254, "total_tokens": 1314 }
}

The usage object is how you meter cost. On a token-metered gateway, total_tokens is the number that matters; a short summarization runs around 1.3K tokens.

Common parameters and models #

FieldTypeNotes
modelstringRequired. An id from your gateway's catalog.
messagesarrayRequired. {role, content} objects; roles are system, user, assistant.
max_tokensintegerCaps the completion length. A keyword line plus two short paragraphs fits in 1024.
temperaturenumberOptional. Lower (0 to 0.4) for steady, repeatable summaries.
streambooleanOptional. Keep it false for batch work; set true only for a live, typing-as-it-goes UI.
Picking a model Turning a few free-text responses into a short, cited summary is a small job. A cheap, fast model (a Haiku-class or Nova-class model, for instance) is the right default. Save the large models for genuinely hard reasoning; here they cost more tokens for no gain.

Status codes #

Because it is OpenAI-compatible, errors arrive as HTTP status codes with a JSON body. Handle them the way you would the OpenAI API; the cookbook below has the fixes.

StatusMeaningWhat to do
401 / 403Key missing, wrong, or wrong headerCheck the key and the auth header. Don't retry.
404Unknown model or wrong pathVerify the model id and that the base URL ends in /v1.
429Rate or quota limitBack off and retry. A cache keeps you under it.
5xxGateway or upstream errorRetry a few times, then skip that record so one failure does not stop the batch.

Copy, paste, adapt

Snippets that run #

The same call in four flavors. Each takes a list of free-text responses and returns a short, cited summary. The base URL, key, and model come from the environment, never hard-coded.

Auth, in code These use the standard bearer token (the SDK sends it from api_key; the raw examples set the Authorization header). If your gateway wants x-api-key instead, swap that one header. 
# pip install openai
import os
from openai import OpenAI

client = OpenAI(
    base_url=os.environ["LLM_BASE_URL"],   # https://<host>/v1
    api_key=os.environ["LLM_API_KEY"],     # sent as Authorization: Bearer
)

def summarize(responses):
    numbered = "\n".join(f"[{i}] {r}" for i, r in enumerate(responses, 1))
    resp = client.chat.completions.create(
        model=os.environ.get("LLM_MODEL", "your-model-id"),
        max_tokens=1024,
        messages=[
            {"role": "system", "content": "You summarize free-text responses."},
            {"role": "user",   "content": numbered},
            {"role": "user",   "content": "List the main themes, each with the [n] that support it."},
        ],
    )
    print("tokens:", resp.usage.total_tokens)
    return resp.choices[0].message.content
# pip install requests, no SDK, pure HTTP
import os, requests

def summarize(responses):
    numbered = "\n".join(f"[{i}] {r}" for i, r in enumerate(responses, 1))
    r = requests.post(
        os.environ["LLM_BASE_URL"] + "/chat/completions",
        headers={
            "Authorization": "Bearer " + os.environ["LLM_API_KEY"],
            "Content-Type": "application/json",
        },
        json={
            "model": os.environ.get("LLM_MODEL", "your-model-id"),
            "max_tokens": 1024,
            "messages": [
                {"role": "system", "content": "You summarize free-text responses."},
                {"role": "user",   "content": numbered},
                {"role": "user",   "content": "List the main themes, each with the [n] that support it."},
            ],
        },
        timeout=60,
    )
    r.raise_for_status()
    return r.json()["choices"][0]["message"]["content"]
// Node 18+ has fetch built in. No dependencies.
async function summarize(responses) {
  const numbered = responses.map((r, i) => `[${i + 1}] ${r}`).join("\n");
  const res = await fetch(process.env.LLM_BASE_URL + "/chat/completions", {
    method: "POST",
    headers: {
      "Authorization": "Bearer " + process.env.LLM_API_KEY,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      model: process.env.LLM_MODEL ?? "your-model-id",
      max_tokens: 1024,
      messages: [
        { role: "system", content: "You summarize free-text responses." },
        { role: "user",   content: numbered },
        { role: "user",   content: "List the main themes, each with the [n] that support it." },
      ],
    }),
  });
  if (!res.ok) throw new Error(`Gateway ${res.status}`);
  return (await res.json()).choices[0].message.content;
}
# Smoke-test the gateway from any POSIX shell (macOS, Linux, Git Bash).
curl "$LLM_BASE_URL/chat/completions" \
  -H "Authorization: Bearer $LLM_API_KEY" \
  -H "Content-Type: application/json" \

    ]
  }'
On Windows The cURL tab uses POSIX quoting and $VAR expansion. In Windows cmd or PowerShell the syntax differs; use Git Bash or WSL, or one of the Python or Node snippets.

When it does not work

Common-errors cookbook #

Most failures against an OpenAI-compatible gateway are auth or path problems, not your code. The real ones, with the fix:

SymptomLikely causeFix
401 / 403 on every request.Key missing, wrong, or expired, or the gateway wants a different auth header than you are sending.Check the key. If a bearer token 401s, try an x-api-key header (or the reverse). Your gateway's docs say which. Don't retry a 401.
Works with curl but not the SDK, or the reverse.The two send different headers, or a strict gateway or WAF rejects an unexpected one (the SDK adds its own User-Agent).Match the header your gateway documents. If a WAF is in the way, a plain requests or curl call with only the headers you set is the cleanest test.
404 on POST /chat/completions.The base URL is missing /v1, or the path is doubled (the base URL already ends in /chat/completions).Base URL ends in /v1. The SDK appends /chat/completions for you; in raw HTTP you append it once, so never put it in the env var.
404 model_not_found though auth works.The model id is not in this gateway's catalog, or it is misspelled.List the live ids with GET {BASE_URL}/models and set LLM_MODEL; do not hard-code.
OpenAIError: the api_key client option must be set.api_key is None and no OPENAI_API_KEY is set.Pass a non-empty api_key (the real key, or a placeholder if your gateway reads a different header).
stream: true hangs or returns nothing usable.Streaming returns server-sent-event chunks that use delta, not one message body; some proxies buffer them.Keep stream: false for batch work and read choices[0].message.content. For a live UI, iterate the SSE delta.
Intermittent 429 or 5xx mid-batch.429 is a rate or quota limit; 5xx is a gateway or upstream error. Neither is an auth or syntax problem.Back off and retry on 429 / 5xx; do not retry 401 / 403 / 404. A warm cache cuts how often you hit 429.
