UCSB use case

Sabbatical requests

For Academic personnel analysts

You have

Faculty sabbatical request packets (free-text justifications and attachments)

↓
You want

A preliminary completeness check per request and a clean extract for the tracker

How it works

The same pattern the whole guide teaches, applied to sabbatical request packets: scripts gather the packets and the required-field policy, the model is brought in only to read each free-text justification and flag what is missing, and an analyst approves before anything reaches the tracker.

  1. 1
    You

    Get the packets into a table

    Export the sabbatical request packets to a spreadsheet: one row per request, an id for each request, the free-text justification, and a column noting which attachments came in. This is the only step that touches the raw, possibly protected, packet text. Sabbatical packets are P3 data, so keep the whole pipeline on the gateway and out of any general-purpose chat.

  2. 2
    Script

    Let code load the policy and the requests

    A script loads the requests and your required-field policy (the list of fields and attachments every packet must include), then adds a stable request_id to every row so each justification can be cited later. The policy lives in code, where it is exact and repeatable, so the same checklist is applied to every request the same way.

    Prompt to paste 
    Write a Colab cell that uses pandas to load my sabbatical requests CSV into a DataFrame called df and a separate required-field policy list (the fields and attachments every packet must include, e.g. {leave_dates, project_plan, funding_status, chair_endorsement}). Lowercase and snake_case the column names, drop fully empty rows, and add an integer request_id to every row so each justification can be cited later. Print df.head() and the policy list so I can confirm both loaded. Do not send anything to a model in this cell.
  3. 3
    AI, the one hard part

    Check each packet against policy, with citations

    The model reads each free-text justification and returns a bulleted list of what is missing against the required-field policy, plus the fields ready to drop into the tracker, citing the request_id it judged. Justification text in, a completeness check and a clean extract out. It never sees the faculty member's name.

    Prompt to paste 
    Write a Colab cell that, for each request, sends only the request_id, the justification text, and the required-field policy list to the LLM Sandbox gateway (an OpenAI-compatible endpoint; read the base URL and API key from Colab Secrets, never hardcode them). Ask the model to return JSON with two keys: missing (a bulleted list of required fields or attachments not satisfied by this packet) and extract (the tracker fields it could find, e.g. {leave_dates, project_plan_summary, funding_status}). Tell it to cite the request_id it read in square brackets like [7], name no individuals, and reply with an empty missing list when the packet satisfies every policy item. Collect the results into a dict keyed by request_id. Never paste raw packet text into this chat.
  4. 4
    A person

    Review before it reaches the tracker

    The analyst reads the flagged gaps against each cited packet, corrects anything the model missed or overcalled, and only then acts. This is a preliminary check, not a decision: the human stays at the gate, and the model never has the last word on completeness.

  5. 5
    Output

    Out comes the check and the extract

    A per-request completeness summary (what is missing, with the requesting analyst's edits) plus a clean row of tracker fields for each packet, ready to paste or import into your sabbatical tracker.

    Prompt to paste 
    Write a Colab cell that builds, for each request_id, a short completeness report with the missing-items list on top and the extracted tracker fields below, rendering each [request_id] as a footnote linking back to that packet. Also assemble all the extracts into one tidy DataFrame (one row per request) and save it as tracker_extract.csv I can download. Save each report as its own file in a reports/ folder and print the list of files written. Make the report format {Markdown or .docx}.
