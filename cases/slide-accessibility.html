UCSB use case

Slide accessibility

For Accessibility and media teams

You have

Dense presentation decks of graphs and equations that need descriptive text

↓
You want

Per-slide, WCAG-compliant descriptions published with the talk

How it works

The same pattern, applied to slide decks: a script slices each deck into per-slide images and a multimodal model writes one screen-reader description per slide, with citations back to the exact slide and a person at the gate.

  1. 1
    You

    Gather the decks

    Collect the presentation files (PowerPoint, PDF, or Google Slides exports) into one folder and note, for each deck, the talk title and where the descriptions will be published. This is the only step that touches the raw source material, and the decks stay on your side.

  2. 2
    Script

    Slice each deck into per-slide images

    A script renders every deck to one image per slide and builds a manifest: a deck_id, an integer slide_id, the rendered image path, and any speaker-notes or on-slide text it can read. Slicing, ordering, and bookkeeping stay in code, where they are exact and repeatable, never in the model.

    Prompt to paste 
    Write a Colab cell that walks a decks/ folder and, for each file, renders one PNG per slide into slides/{deck_id}/{slide_id}.png (use pdf2image for PDFs and convert PowerPoint with LibreOffice in headless mode first). Build a pandas DataFrame manifest with one row per slide holding deck_id, an integer slide_id starting at 1, the image path, and any extractable on-slide text or speaker notes. Save the manifest to slides_manifest.csv and print its head. Do all the slicing and bookkeeping in code, never in a model.
  3. 3
    AI, the one hard part

    Write the screen-reader description per slide

    The multimodal model looks at each slide image and writes a concise, WCAG-aligned description: what the graph or equation conveys, not a pixel-by-pixel readout. Image in, cited description out. It is the one thing a script cannot do, and it never sees who created the deck.

    Prompt to paste 
    Write a Colab cell that, for each row in the manifest, sends that slide's image (base64) plus its slide_id and any extracted on-slide text to the LLM Sandbox gateway (an OpenAI-compatible, multimodal endpoint; read the base URL and API key from Colab Secrets, never hardcode them). Ask for a WCAG-aligned screen-reader description of at most {N, e.g. 2} sentences that conveys what a graph or equation means rather than listing pixels, tags the slide it describes in square brackets like [deck_id:slide_id], and replies "needs human description" when the slide is too dense or ambiguous to caption safely. Collect the results into the DataFrame keyed by deck_id and slide_id. Never paste raw deck files into this chat.
  4. 4
    A person

    Spot-check and approve

    An accessibility reviewer reads each description against its slide, rewrites anything that is wrong or thin, and clears the ones the model flagged as "needs human description". The human stays at the gate; the model never has the last word on what ships.

  5. 5
    Output

    Out come the published descriptions

    Per-slide, WCAG-compliant descriptions attached to the deck and published alongside the talk: alt text, a long-description file, or accessible-PDF tags, each tied back to the slide it came from.

    Prompt to paste 
    Write a Colab cell that, from the approved DataFrame, writes the descriptions back out per deck: one descriptions file per deck (slide_id, then its approved text) and a combined long-description page in {Markdown or HTML} with each description under a heading for its slide. Save everything into a published/ folder I can download, keep slides in numeric order, and print the list of files written.
