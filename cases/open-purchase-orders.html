UCSB use case

Open purchase orders and receiving

For Procurement, finance, and department staff

You have

Purchase-order data: open POs, days open, financial exposure, missing receiving documents

↓
You want

A campus-wide aging dashboard with an exposure narrative, and automated department-level follow-ups

How it works

The same pattern the whole guide teaches, applied to purchase-order data: scripts compute every aging bucket and exposure figure, the model is brought in only to write the narrative over those numbers, and a person approves before any follow-up goes out.

  1. 1
    You

    Get the PO data out

    Export the open purchase orders to a spreadsheet: one row per PO, with the PO number, department, vendor, amount, open date, and a flag for whether receiving documents are missing. This is the only step that touches the raw, possibly protected, financial data.

  2. 2
    Script

    Let code do the aging and exposure

    A script computes days open for every PO, sorts them into aging buckets (0 to 30, 31 to 60, 61 to 90, over 90), totals the dollar exposure per bucket and per department, and counts POs with missing receiving documents. Math stays in code, where it is exact and repeatable, never in the model.

    Prompt to paste 
    Write a Colab cell that uses pandas to load my CSV of open purchase orders into a DataFrame called df. Clean it: lowercase and snake_case the column names, parse open_date as a datetime, and drop fully empty rows. Compute days_open as today minus open_date, then add an aging_bucket column with the cuts {bucket edges, e.g. 30/60/90 days}. Build a summary table grouped by {the unit you want one report per, e.g. department} with the count of POs, total dollar exposure, exposure in each aging bucket, and the count of POs where missing_receiving is true, rounding dollars to whole numbers. Print df.head() and the summary table. Do all the math in pandas, never in a model.
  3. 3
    AI, the one hard part

    Write the exposure narrative over the numbers

    There is no free text to read here, so the model does the one thing the script cannot: it turns the computed aging and exposure numbers into a short executive narrative with recommended actions. Numbers in, plain-language brief out. It sees only the aggregate figures, never PO-level records or vendor names.

    Prompt to paste 
    Write a Colab cell that sends only the computed summary numbers (per {department}: total open POs, total exposure, exposure by aging bucket, and count of POs missing receiving docs) to the LLM Sandbox gateway (an OpenAI-compatible endpoint; read the base URL and API key from Colab Secrets, never hardcode them). Ask for a {N, e.g. 4}-sentence executive narrative that calls out where exposure is concentrated and the oldest aging, plus {M, e.g. 3} recommended actions, in neutral finance language. Tell it to reference figures exactly as given and invent no numbers. Collect the result keyed by {department}. Never paste raw PO records into this chat.
  4. 4
    A person

    Verify the figures and approve

    Finance staff check the narrative against the computed table, confirm the exposure and aging numbers are right, fix anything off, and only then release it. The human stays at the gate; the model never has the last word, and no automated follow-up sends before sign-off.

  5. 5
    Output

    Out comes the dashboard and the follow-ups

    A campus-wide aging dashboard with the exposure narrative on top, plus per-department follow-up emails to receivers asking them to close out old POs, each gated behind the finance verification step.

    Prompt to paste 
    Write a Colab cell that builds one report per {department} with the aging and exposure summary table first and the approved narrative below, then drafts a follow-up email to that department's receiver listing its oldest POs and the ones missing receiving docs. Save each report and its draft email as files in a reports/ folder I can download, mark every draft as needing finance sign-off before sending, and print the list of files written. Make the format {Markdown or .docx}.
