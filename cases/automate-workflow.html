UCSB use case

Automate a repetitive workflow

For Non-coders with a manual, repetitive process

Used AI to write the code (often Google Apps Script) that automates a repetitive process end to end, with a human approval gate. The same approach handled quarterly course scheduling, room bookings, graduate onboarding emails, mapping inventory into a rigid system template, and consolidating leads from many tools into one live sheet. Real data stays out of the model; the AI writes the script, you run it.

How it works

The same pattern, applied to a manual chore you repeat every week: AI writes the script, the script does the exact work on your side, and you approve every run before it acts.

  1. 1
    You

    Describe the workflow, step by step

    Write down the chore exactly as you do it by hand: what you open, what you copy, where it goes, and the rule for each decision (for example, "if the room is already booked, flag the row; otherwise add the event"). The clearer the steps, the better the script. Nothing protected leaves your machine here; you are only describing the process.

  2. 2
    AI

    Have AI write the script

    Hand your written steps to the AI and ask it to write a Google Apps Script that does the whole chore for you. It writes the code; it never sees your real records. You paste the script into the Apps Script editor (Extensions, Apps Script) attached to your sheet or doc, and the script runs on your side against your own data.

    Prompt to paste 
    Write a Google Apps Script that automates this workflow on my own Google Sheet: {describe the chore, e.g. read each pending room request from the "Requests" tab and create a calendar event}. Read inputs from the sheet (or the bound Google service, e.g. Calendar, Gmail, Drive) using the built-in Apps Script services, never from a hardcoded copy of the data. For every action, decide using this rule: {your rule, e.g. skip and flag any row whose time overlaps an existing event}. Do not send any row data to an external model; all logic runs in the script. Log every action to a "Log" tab with a timestamp, and put everything behind a {dry-run flag or confirmation} so it shows me what it WOULD do before it does anything. Add clear comments and a runnable main() function I can review.
  3. 3
    A person

    Review, dry-run, then run

    Read the script, run it once in dry-run mode, and check the log of what it would do against a few rows you know. When it looks right, flip off the dry-run and run it for real (or set a time-based trigger so it runs on a schedule). The human stays at the gate: the script only does what you have read and approved.
