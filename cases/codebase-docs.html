UCSB use case

Undocumented codebases

For IT and application developers

You have

Legacy application source code with no technical documentation

↓
You want

A structured technical document: system overview, file map, and feature breakdowns

How it works

The same pattern, applied to source code: scripts gather and chunk the files, the model is brought in only to synthesize a structured technical doc with traceable mappings, and a developer signs off before anything is published.

  1. 1
    You

    Point at the codebase

    Get a clean copy of the legacy source you want documented: clone the repo or drop the project folder into Colab. Decide the scope (one service, one module, or the whole app) and which file types matter (the languages, config, and templates). This is the only step that touches the raw, possibly sensitive, source.

  2. 2
    Script

    Let code gather and chunk the files

    A script walks the project tree, keeps the relevant files, skips the noise (vendored dependencies, build artifacts, lockfiles, .git), and splits each file into numbered chunks small enough to send. Every chunk carries a stable id and its file path, so anything the model writes can be traced back to an exact location. The inventory and the chunking are exact and repeatable, so they stay in code, never in the model.

    Prompt to paste 
    Write a Colab cell that uses pathlib to walk {the project folder, e.g. ./legacy-app}, keep only source files with extensions in {the set you care about, e.g. .py, .js, .sql, .html}, and skip directories like node_modules, vendor, dist, build, and .git. For each kept file, read its text and split it into chunks of about {N lines, e.g. 200} lines, and record each chunk as a dict with chunk_id (an incrementing integer), file_path, start_line, end_line, and the chunk text. Collect them into a list called chunks, and also build a files table (path, language guessed from extension, line count). Print the files table and len(chunks). Do all the file walking and chunking in code, never in a model.
  3. 3
    AI, the one hard part

    Synthesize the structured doc, with traceable mappings

    The model reads the numbered code chunks and writes the parts a script cannot: a system overview, a file map, and per-feature breakdowns, each claim citing the chunk_ids it drew from so a developer can jump to the exact file and lines. It also returns an open-questions list where the code is ambiguous, which feeds the refinement loop. Numbered chunks in, cited document out. It only ever sees code text and ids, no credentials or secrets.

    Prompt to paste 
    Write a Colab cell that sends the code chunks to the LLM Sandbox gateway (an OpenAI-compatible endpoint; read the base URL and API key from Colab Secrets, never hardcode them), passing only chunk_id, file_path, and chunk text, batched so each request stays under the model's context limit. Ask it to return JSON with three sections, a {system overview}, a {file map} (one line per file describing its responsibility), and {feature breakdowns}, where every statement cites the chunk_ids it used in square brackets like [42], plus an open_questions list for anything the code does not make clear. Before sending, strip obvious secrets (lines matching {patterns like API_KEY, PASSWORD, token=}). Merge the batched results into one structured doc. Never paste raw source into this chat.
  4. 4
    A person

    Run the open-questions loop, then approve

    A developer reads the draft, follows the cited chunk_ids back to the real code to confirm each mapping, and answers the open-questions list (or feeds the answers back for another pass). When the overview, file map, and feature breakdowns match the code, the developer signs off. The human stays at the gate; the model never has the last word on what the system actually does.

  5. 5
    Output

    Out comes the technical document

    A single structured technical document, system overview, file map, and feature breakdowns, with every claim linked back to the source it came from, ready to drop into the repo's docs or a wiki for the next developer who inherits the code.

    Prompt to paste 
    Write a Colab cell that renders the approved doc as Markdown: the system overview first, then the file map as a table (file_path, responsibility), then a section per feature, turning each [chunk_id] citation into a link or footnote that shows the file_path and line range it points to. Append the resolved open-questions as a short notes section. Save it as {docs/ARCHITECTURE.md or the path your repo uses} so I can download it, and print the file path written.
