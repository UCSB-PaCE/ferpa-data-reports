UCSB use case

Advising traffic

For Advising directors and student affairs

You have

Monthly walk-in visit logs from a queue system (after PII is stripped)

↓
You want

A readable report of peak hours, station load, and top reasons for visits

How it works

The same pattern the whole guide teaches, applied to advising queue logs: scripts own the counts (peak hours, station load), the model is brought in only to read the reason-for-visit text, and a person approves before the report goes out.

  1. 1
    You

    Get the visit logs out

    Export the month of walk-in visits from your queue system to a spreadsheet: one row per visit, with the date, time in, station, wait time, and the reason-for-visit text. Strip names and any identifiers first. This is the only step that touches the raw, possibly protected, data.

  2. 2
    Script

    Let code do the counting

    A script computes everything that is pure arithmetic: visits per hour of the day, visits and average wait per station, and day-of-week volume. Counting and peaks stay in code, where they are exact and repeatable, never in the model.

    Prompt to paste 
    Write a Colab cell that uses pandas to load my CSV (start with advising_traffic_sample.csv) into a DataFrame called df. Clean it: lowercase and snake_case the column names, parse date and time_in, drop fully empty rows, and add an integer visit_id_num to every row so each reason can be cited later. Then derive an hour column from time_in and build three summary tables: visits per hour of day, visits and mean wait_min per {the unit you want load per, e.g. station}, and visits per day of week. Round the means to one decimal. Print df.head() and the three tables. Do all the math in pandas, never in a model.
  3. 3
    AI, the one hard part

    Categorize the reasons and explain the spikes

    The model reads the numbered reason-for-visit text, sorts each visit into a named category (drop or withdrawal, prerequisites, petitions, transfer credit, and the like), and writes a short narrative that explains the busy stretches using the categories it found. Free text in, named categories plus a cited narrative out. It never sees who came in.

    Prompt to paste 
    Write a Colab cell that sends only the visit_id_num and reason_text (never names or any other field) to the LLM Sandbox gateway (an OpenAI-compatible endpoint; read the base URL and API key from Colab Secrets, never hardcode them). Ask the model to assign each reason to one of {a short fixed list of categories, e.g. drop/withdrawal, prerequisites, petitions, transfer credit, graduation check}, returning a dict of visit_id_num to category, plus a {N, e.g. 4}-sentence narrative explaining the peak hours and station load in terms of those categories, citing the visit_id_nums it leaned on in square brackets like [10293]. Tell it to use "other" when a reason does not fit and to name no individuals. Send the reasons in batches, collect the category map, and merge it back onto df. Never paste raw records into this chat.
  4. 4
    A person

    Read it and approve

    You spot-check the assigned categories and the narrative against the cited visits, fix any miscategorized reasons, and only then release it. The human stays at the gate; the model never has the last word on staffing.

  5. 5
    Output

    Out comes the report

    A readable report of peak hours, station load, and the top reasons for visits, with the narrative explaining the spikes, ready to hand to whoever sets the advising schedule.

    Prompt to paste 
    Write a Colab cell that builds one report with the peak-hours and per-station tables first (render them as simple bar-style tables), a ranked list of the top reason categories with counts, and the cited narrative below, rendering each [visit_id_num] as a footnote that shows the verbatim reason. Save it as a file in a reports/ folder I can download, and print the path. Make the format {Markdown or .docx}.
