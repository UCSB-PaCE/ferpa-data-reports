Any organization use case

Meeting notes

For Any team that runs recurring meetings

You have

A folder of raw meeting notes and transcripts across a quarter

↓
You want

A decisions-and-actions digest with owners and open items

How it works

The same pattern the whole guide teaches, applied to a quarter of raw meeting notes: a script collects and numbers every note, the model is brought in only to pull out the decisions, action items, and owners, and a person approves the digest before it goes out.

  1. 1
    You

    Get the notes into one sheet

    Gather the quarter's meeting notes and transcripts into a single spreadsheet: one row per note or meeting, with the meeting date, the topic, and the full note text in a free-text column. This is the only step that touches the raw notes, which can name people and carry sensitive detail.

  2. 2
    Script

    Let code collect and number the notes

    A script loads the sheet, absorbs its inconsistent shape, and stamps every note with an id so each extracted item can be cited back to its source. It also counts the simple, exact things (notes per topic, notes per meeting date) that never need a model. Shaping and counting stay in code, where they are repeatable.

    Prompt to paste 
    Write a Colab cell that uses pandas to load my CSV (start with meeting_notes_sample.csv) into a DataFrame called df. Clean it: lowercase and snake_case the column names, parse meeting_date as a date, drop rows where notes_text is empty, and add an integer note_id to every row so each note can be cited later. Then build a small summary table of note counts grouped by {the unit you want one digest per, e.g. topic} sorted by most recent meeting_date. Print df.head() and the summary table. Do all the shaping and counting in pandas, never in a model.
  3. 3
    AI, the one hard part

    Extract decisions, actions, and owners

    The model reads each numbered note and pulls out the structured items a script cannot: the decisions made, the action items, who owns each one, and which items are still open. Every extracted item cites the note it came from. Numbered notes in, a structured digest out, with no name ever sent that is not already in the note text you choose to include.

    Prompt to paste 
    Write a Colab cell that, for each {group, e.g. topic}, gathers that group's notes and sends only the note_id and notes_text to the LLM Sandbox gateway (an OpenAI-compatible endpoint; read the base URL and API key from Colab Secrets, never hardcode them). Ask it to return JSON with three lists, decisions, action_items, and open_items, where each item is an object with a short text field, an owner field (or "unassigned"), and a sources field listing the note_ids it used in square brackets like [12]. Tell it to invent nothing not present in the notes and to leave a list empty when there is nothing to report. Collect the results into a dict keyed by group. Never paste raw notes into this chat.
  4. 4
    A person

    Check it against the source notes

    You open each item, follow its cited note_id back to the original note, confirm the decision and owner are right, and fix anything the model misread before the digest goes out. The human stays at the gate; the model never has the last word on who owns what.

  5. 5
    Output

    Out comes the digest

    A decisions-and-actions digest per topic (or one for the whole quarter): the decisions made, the action items with owners, and a clearly flagged list of open items still needing an owner, each line traceable back to the note it came from.

    Prompt to paste 
    Write a Colab cell that builds one digest per {group} with three sections, Decisions, Action items (with owners), and Open items, rendering each [note_id] as a footnote that shows the verbatim note and its meeting_date. Put unassigned or open items in their own clearly labeled section so nothing falls through. Save each digest as its own file in a digests/ folder I can download, and print the list of files written. Make the format {Markdown or .docx}.
