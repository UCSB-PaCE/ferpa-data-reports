UCSB use case

Help-desk inbox

For System owners and support leads

You have

Three years of help-desk emails from a delegate inbox

↓
You want

A ranked summary of recurring pain points to drive a redesign

How it works

The same pattern the whole guide teaches, applied to a delegate inbox: scripts pull and de-duplicate the thread history, the model is brought in only to cluster the free-text emails into named, frequency-ranked pain points, and a person approves the priorities before they drive a redesign.

  1. 1
    You

    Get the emails out

    Export three years of messages from the delegate help-desk inbox to a spreadsheet: one row per email, with the date, any existing category tag, the resolution time, and the email body. This is the only step that touches the raw inbox, where addresses and names may live, so strip or leave behind anything you do not need before it goes any further.

  2. 2
    Script

    Let code clean and count

    A script absorbs the export's shape, removes near-duplicate emails (the same issue reported many times, quoted reply chains, autoreplies), and counts the raw volume per existing category and per month. De-duplication and counting stay in code, where they are exact and repeatable, never in the model.

    Prompt to paste 
    Write a Colab cell that uses pandas to load my CSV (start with helpdesk_emails_sample.csv) into a DataFrame called df. Clean it: lowercase and snake_case the column names, parse date to a datetime, drop fully empty rows, and add an integer ticket_ref to every row so each email can be cited later (or reuse ticket_id if present). De-duplicate the free text: normalize email_body (strip quoted reply chains and signatures, collapse whitespace, lowercase) into a temp column and drop exact and near-exact repeats, keeping a count of how many times each distinct message appeared. Then print df.head(), the row counts before and after de-duplication, and a table of volume grouped by {an existing field, e.g. category} and by month. Do all the counting in pandas, never in a model.
  3. 3
    AI, the one hard part

    Cluster the emails into named pain points

    The model reads the de-duplicated, numbered emails and groups them into a handful of named, recurring pain points, each one citing the emails it drew from. Free text in, a clustered list of themes out. It sees the message bodies and their ids only, never who sent them.

    Prompt to paste 
    Write a Colab cell that sends only the ticket_ref and the de-duplicated email text (never sender names or addresses) to the LLM Sandbox gateway (an OpenAI-compatible endpoint; read the base URL and API key from Colab Secrets, never hardcode them). Batch the emails so each request stays within the model's context, and ask it to cluster the messages into at most {N, e.g. 8} named, distinct pain points. For each cluster return a short label, a one-sentence description, and the list of ticket_refs it used in square brackets like [HD-4823]. Have the script tally the cited ticket_refs (weighting by the duplicate count from the previous step) so the clusters can be ranked by frequency, and merge near-identical labels across batches. Name no individuals, and reply "not enough emails" for any cluster under {min, e.g. 5} messages. Never paste raw emails into this chat.
  4. 4
    A person

    Read it and approve the priorities

    You spot-check each named pain point against its cited emails, merge or split clusters that are off, and confirm the ranking reflects what the team actually feels before it shapes a redesign. The human stays at the gate; the model never sets the priorities on its own.

  5. 5
    Output

    Out comes the ranked summary

    A ranked summary of recurring pain points, most frequent first, each with its volume, its trend over the three years, and the source emails behind it, ready to drive a redesign and to brief the people who own the system.

    Prompt to paste 
    Write a Colab cell that builds one ranked report of the pain points: each named cluster as a section ordered by frequency, showing its volume, its share of total emails, a small month-by-month trend from the counts computed earlier, and the cited emails as footnotes that show the verbatim message behind each [ticket_ref]. Save it as a single file I can download and print the path. Make the format {Markdown or .docx}.
