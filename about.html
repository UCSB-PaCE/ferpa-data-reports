About

About this guide.

This is a working write-up of something built at UCSB Professional & Continuing Education and then generalized: a repeatable way to turn data into reports, with a person in charge and AI used only where it helps. It is meant to be copied. Take the parts that fit your work and leave the rest.

Who made it

Jonathan Engeln Lambinet-Lacson

I work at UCSB Professional & Continuing Education, where this pipeline started as a way to turn confidential course evaluations into reports our instructors could actually use. The interesting part was never a single clever trick. It was treating the work as a process to improve: writing scripts for the parts a computer should own, bringing in a language model only where it earned its place, and keeping a person at the gate the whole time. Once it was solid, the shape turned out to fit almost any data, so I wrote it down.

My hope for this guide is simple. Plenty of smart people at a university are one good walkthrough away from doing this themselves. If the no-code path turns a few of them into people who can take their own data and produce a trustworthy report, it did its job.

Full transparency

How this guide was made

This site is itself an example of the thing it describes. I set the goal and the guardrails and approved every change; a language model, working through UCSB's LLM Sandbox, did the drafting and a lot of the heavy lifting. The same human-at-the-gate rule the pipeline uses is the rule that produced these pages. If you find something off, that is the check working, tell me and I will fix it.

The assistant in the corner

The "Ask the assistant" button opens a small chat backed by a bot published on UCSB's LLM Sandbox, grounded only on this site. It is the lesson applied to itself: a small, cheap model, a tight scope, retrieval limited to this content, and citations on. Its configuration:

The key-hiding trick (for the developers reading)

The chat widget sits on a public page, yet its API key is nowhere in the source. The browser calls a small Cloudflare Worker that holds the key as a server-side secret, allows only this origin, rate-limits, and forwards the request with the key attached. That is the standard way to put a keyed API behind a public site, and the Worker code is in the repository.
