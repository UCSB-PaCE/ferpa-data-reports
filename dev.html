For developers

Build it with code

The job is a pipeline of six stages. Scripts do the steady work, but the model is wired into the run itself: when a file does not fit, it writes a fresh intake layer; when an identity will not match, it writes the lookup; and it turns free text into a cited summary. A person approves what ships. This page builds each stage with code you can lift, and points to the gateway page for the model call.

The idea that transfers

Six stages, three roles #

Underneath any version of this is the same six-stage spine. Scripts handle the repeatable work. The model is called at runtime wherever the pipeline has to adapt or judge, so a changed file or a stubborn match heals instead of breaking, and a person approves the result. That mix, AI where it earns its place and plain code everywhere else, is what makes it both capable and safe.

This is a self-healing pipeline The model is part of the run, not just a tool you used to build it. When a file arrives in the wrong shape, the pipeline asks the model to read the structure and write a fresh intake layer, so a format change adapts instead of breaking. The same move resolves an identity a script cannot, and stage 4 summarizes the free text. Scripts do everything repeatable; the model steps in to adapt or judge; a person approves. It adds up to more model than you might think, see how much AI is really in here.

What "your data" can be #

The pattern fits any record with some structured fields, some free text a human wrote, and a natural subject to report on. A few examples of the same machine, aimed somewhere new:

Structured columns drive the charts. Free text drives the model. Reconciliation ties them to a subject. A cache makes it cheap. A template makes it shareable. Everything else is dataset-specific glue.

Stage 1 script + AI

Ingest #

Read whatever you were handed into a dataframe. The catch is that the next file is never quite the same: a renamed column, a different export format, an extra header row. Instead of breaking on it, the pipeline checks the shape, and when it does not fit, calls the model to write a fresh intake layer for that file. A format change becomes an adaptation, not an outage. This is the self-healing part, and it is the model doing real work inside the run, not just helping you build it.

import pandas as pd

df = pd.read_csv("export.csv")               # or read_excel, or a SIS dump

if not fits_expected_schema(df):            # columns are off, format changed...
    layer = write_intake_layer(schema_of(df))   # the model writes the fix...
    df = layer(df)                           # ...as a script you read and keep

The model sees the shape, the column names and types, not your records, and what it returns is a script you review and save. The next time that format shows up, the layer already exists and no model call is needed.

Stage 2 script

Standardize #

Rename columns to names your code expects, validate types, and coerce scales. Work on a copy and leave the source export read-only, so you can always re-derive and never corrupt the original.

clean = df.rename(columns={"Instructor Name": "subject", "Comments": "text"})
clean["text"] = clean["text"].fillna("").str.strip()
clean = clean[clean["text"] != ""]      # drop empty responses

Stage 3 script

Reconcile identity #

Tie each row to a real subject, a person, a course, a cohort, against a roster of truth. Fuzzy matching handles typos and aliases; a score threshold keeps it from guessing. When a row will not match, that is a good place to ask the model for help: have it draft the lookup that resolves the identity, an LDAP query against your directory, say, so no one is hand-matching hundreds of names. More of these in how much AI is in here.

from thefuzz import process

def match(name, roster):
    best, score = process.extractOne(name, roster)
    return best if score >= 90 else None   # unmatched rows get reviewed, not guessed

Stage 4 AI (cached)

AI insight #

This is the one stage where the model reads your records' actual text. The intake and reconcile calls work on shapes and identifiers; here the free text itself goes to the model. Send it numbered, and ask for a short summary that cites the comments it draws on. The call itself, base URL, key, snippets, lives on the gateway page; what matters here is what you send and that you cache the result.

Send the text, not the people Names, ids, and grades stay in your structured layer. Number the comments so the summary can cite [1] and [3] without the model ever seeing who wrote them. The citations are reattached to the verbatim text on your side, after the model answers. 
# summarize() is the gateway call from the gateway page
key = (subject, period, prompt_version)
insight = cache.get(key)
if insight is None:
    insight = summarize(texts)     # the one billed call
    cache[key] = insight           # pay for it once

Stage 5 script

Render the report #

Combine the structured columns (which become charts) with the cached insight (the words) into a per-subject report. Use whatever you know: an HTML template, a charting library, a PDF tool like reportlab. The format is glue; the content is already made.

build_report(
    subject=subject,
    charts=charts_from(clean),     # structured columns -> charts
    insight=insight,               # the cached, cited summary
)

Stage 6 human

Distribute #

Nothing ships on its own. A person reads the report and approves it; then a script uploads, emails, or files it. The model never has release authority. That one rule is both your quality control and your compliance control, and it is the cheapest safeguard you have.

Plain scripts where the work repeats, a model call wherever the run must adapt or judge, a person at the gate. That mix is the design, and it is why a changed file adapts instead of breaking the run.

Count them up

How much AI is really in here #

It is more model than "one call." The pipeline reaches for it at runtime wherever it has to adapt or judge, and again between cycles to get better. Where it earns its place:

Adapt the intake runtime

A file in the wrong shape triggers a model call that writes a fresh intake layer, so the run heals instead of breaking (stage 1).

Resolve hard matches runtime

When fuzzy matching fails, the model writes the lookup that resolves the identity, an LDAP query against your directory, say (stage 3).

Summarize the text runtime

The one call that reads your records: free text in, a short cited summary out (stage 4).

Improve the process each cycle

The model reads a run's logs and code and proposes the next small, testable change.

The line that keeps it safe Only the stage-4 summary sends your records' text to the model. The intake and reconcile calls ask it to write code, an intake layer, a lookup, which then runs on your side against your own data. A person approves every report. That is how AI ends up woven through the run without you losing hold of the records.

Where the lean lives

Cache the AI work #

The model call is the only thing that costs money, so cache it hard. Key the cache by what makes an analysis identical: the subject, the period, and a prompt version. A re-run then costs nothing, which is what makes the improvement loop below free to use.

import sqlite3

def get_or_analyze(con, subject, period, texts, version="v1"):
    row = con.execute(
        "SELECT insight FROM cache WHERE subject=? AND period=? AND version=?",
        (subject, period, version)).fetchone()
    if row:
        return row[0]                       # warm: 0 tokens
    insight = summarize(texts)             # cold: the one billed call
    con.execute("INSERT INTO cache VALUES (?,?,?,?)", (subject, period, version, insight))
    con.commit()
    return insight

Bump version when you change the prompt and the cache warms again on the next run. Do not cache failed analyses; let them retry cleanly next time.

The method, not a trick

Improve each run #

This is the part that produced the pipeline. The outer loop ships a report every cycle. A faster inner loop, observe, diagnose, improve, verify, keeps making the outer loop leaner and more reliable. Because the cache makes re-runs free, you can replay a fix as often as you like at no token cost.

Self-healinglean · continuous
ObserveRead the run & the code
DiagnoseFind the root cause
ImproveOne small change
VerifyReplay & re-cache
Outer loop: the pipeline that ships the report
Inner loop: the improvement that keeps it lean

The cheapest improvement engine you have is a process that can explain its own last run back to you. Logging row counts, unmatched names, and token use turns each run into something you can learn from, and reading logs and code is exactly the kind of bounded job the same model is good at.

Carry these, forget the rest

Three principles and a checklist #

Build on top, never mutate

The source export is read-only. Every stage produces a new clean layer. You can always re-derive; you can never un-corrupt a source you edited in place.

Cache the expensive work

Pay for new analysis only, never repeated analysis. A good cache key makes "re-run everything" free and the improvement loop usable.

Keep the person at the gate

A model drafts; a human approves. Releasing a report is a human decision, every time. That line is what makes the rest defensible.
